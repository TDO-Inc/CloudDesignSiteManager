## Prompt

I'm building the **TDO Software Client Portal** — a Next.js web app that replaces our ad-hoc Google Drive folder process for collecting website content (copy, photos, brand assets) from clients. This is an internal tool for TDO Software, used by our staff and our dental office clients.

### Your reference documents (read these first)

Before writing any code, read these files in order — they are your source of truth:

1. `CLAUDE.md` — project context, tech stack decisions, resolved questions, current status
2. `tdo-client-portal-spec.md` — full feature specification with data model, feature inventory (core vs extended), V1 pilot scope, integration architecture, and API endpoints
3. `website-templates.md` — detailed template definitions (milestones, file categories, page catalog with content brief fields, Basic vs Standard template differences)
4. `TDO-Client-Portal-UI-UX-Design-Spec.docx` — consolidated UI/UX design document with layout specs, component details, navigation, and brand colors

### Tech stack (already decided — do not change)

- **Framework:** Next.js 15 (App Router)
- **Database:** PostgreSQL
- **ORM:** Drizzle ORM
- **UI:** Tailwind CSS + shadcn/ui
- **Auth (clients):** Custom magic-link with Resend
- **Auth (staff):** Pluggable SSO interface (build the abstraction layer; actual SSO wiring done separately)
- **Email:** Resend, sending from `webadmin@tdo4endo.com`
- **File storage:** Azure Blob Storage (direct upload via SAS URLs)
- **Virus scanning:** Abstraction layer (interface only — implementation plugged in later)
- **Error monitoring:** Sentry
- **Hosting:** Azure (but keep the app portable — no Azure-specific lock-in in the app code)

### What to build — Phase 1 (V1 Pilot)

Build only `[core]` features from the spec. Do NOT build `[extended]` features or Phase 2+ integrations. The V1 scope is:

- Magic-link auth for clients
- Pluggable SSO auth interface for staff (with a dev-mode bypass for local development)
- One service type: Website Design
- 2 starter templates: "Basic" (5-page) and "Standard" (10-page) — seeded from `website-templates.md`
- Client portal: hub-style dashboard, content briefs with auto-save, file uploads with categorization, milestone progress
- Staff portal: project list with filters/search, project detail view, manual project creation, internal notes, activity log
- Email notifications: magic link, file uploaded, content submitted, status changed
- File upload direct to Azure Blob Storage via SAS URLs with virus scan abstraction
- Activity log and email delivery log
- Project archival and hard delete
- Per-project data export (zip of files + briefs)

### Architecture guidance

- Use the data model from Section 4 of the spec exactly. All `jsonb` fields should use Drizzle's jsonb type.
- Template versioning: when a project is created, snapshot the template config into the project so future template edits don't break existing projects.
- Build the virus scanning as an interface/adapter pattern — a `ScanProvider` interface with a `scan(fileBuffer)` method. Include a `NoOpScanProvider` for development that always returns `clean`.
- Build staff SSO as an `AuthProvider` interface. Include a `DevAuthProvider` for local development that auto-authenticates a test staff user.
- File uploads must go directly from the client browser to Azure Blob Storage using SAS URLs generated by the server. Files should never be proxied through the Next.js server.
- Use Next.js App Router with server components by default; client components only where interactivity requires it.
- API routes go under `/app/api/`. Pages go under `/app/(client)/` and `/app/(staff)/`.
- Use route groups to separate client and staff layouts.
- Apply the brand colors from the spec: dark navy `#0F2B3C`, green accent `#1A9E75`.

### Project structure

Set up the project with this structure:

```
src/
  app/
    (client)/          # Client-facing pages (magic-link auth)
    (staff)/           # Staff-facing pages (SSO auth)
    api/               # API routes
  components/
    ui/                # shadcn/ui components
    client/            # Client portal components
    staff/             # Staff portal components
    shared/            # Shared components
  lib/
    db/
      schema.ts        # Drizzle schema (from spec Section 4)
      migrations/      # Drizzle migrations
      seed.ts          # Seed data (service types, templates from website-templates.md)
    auth/
      magic-link.ts    # Magic link implementation
      providers/       # SSO provider interface + dev provider
    storage/
      azure.ts         # Azure Blob Storage SAS URL generation
      scan-providers/  # Virus scan interface + noop provider
    email/
      resend.ts        # Email sending via Resend
      templates/       # Email templates (TDO branded)
    hooks/             # React hooks
    utils/             # Shared utilities
  types/               # TypeScript types
drizzle.config.ts
```

### How to start

1. Read all four reference documents thoroughly.
2. Initialize the Next.js project with TypeScript, Tailwind, and the App Router.
3. Install dependencies: drizzle-orm, pg, @neondatabase/serverless (for Drizzle's PostgreSQL driver), shadcn/ui, resend, @azure/storage-blob, @sentry/nextjs, zod.
4. Set up the Drizzle schema matching the spec's data model.
5. Create the seed script with the Website Design service type and both templates (Basic and Standard) using the exact milestone, file category, and content brief field definitions from `website-templates.md`.
6. Build authentication (magic-link + SSO provider interface).
7. Build the client portal pages.
8. Build the staff portal pages.
9. Build file upload and email notification systems.
10. Wire everything together.

Start by reading the documents and then initializing the project. After each major step, pause and tell me what you've done and what's next.
